# 数据来源与合规规则

本项目默认以不违规、不绕过平台限制、不误导用户为前提。所有后续优化、自动化和数据更新都必须先遵守本文件，再考虑功能效果。

## 当前阶段允许做什么

第一阶段以内容整理和人工核验为主，不做爬虫。

允许：

- 整理官方公开活动页，例如楽天活动日、Amazon 大促、西松屋チラシ、赤ちゃん本舗活动、PayPay/Yahoo 官方活动。
- 阅读公开页面后，用自己的中文解释活动规则、参加方法、注意事项和省钱示例。
- 手动核验少量商品或活动，把来源、更新时间、适用条件写清楚。
- 使用公开可访问的官方页面链接作为来源。
- 写长期攻略，例如尿不湿怎么买、保育园用品怎么买、童装尺码怎么选。
- 把不够确定的信息记录到 `docs/latest-signals.md`，而不是写成确定优惠。

## 当前阶段禁止做什么

禁止：

- 批量爬取 Amazon、楽天、Yahoo、西松屋、赤ちゃん本舗等平台商品数据。
- 高频请求平台页面，例如循环抓取、全天候抓取、几千几万次请求。
- 模拟用户登录、保存用户 Cookie、绕过登录限制。
- 破解验证码、绕过反爬、伪装浏览器指纹。
- 抓取平台全量商品库、评论库、价格库或库存库。
- 把平台页面的大段商品说明、评论、图片直接复制到本站。
- 把未核实、已过期、库存未知的信息写成“正在打折”或“必买”。
- 把抽选、期间限定积分、条件返还写成确定现金折扣。
- 为了提高转化夸大优惠、制造焦虑或诱导囤货。

## 图片使用规则

禁止：

- 批量下载、缓存、转载平台商品图片。
- 批量下载、缓存、转载活动 Banner、チラシ 图片或品牌宣传图。
- 保存或展示用户评论图片、买家晒单图片、头像或其他用户生成图片。
- 为了让页面更真实，把外部平台图片长期存入本站仓库或 CDN。

允许：

- 使用自己制作的图片、图表和说明图。
- 使用官方 API、联盟计划或授权素材中明确允许展示的图片，并遵守尺寸、署名、缓存和跳转要求。
- 在必要范围内使用少量截图作为内部核验或说明材料，并标注来源；公开页面长期展示前必须确认版权和平台规则。

商品图片、品牌图片、活动 Banner 默认不长期保存到本站。需要展示商品时，优先链接到官方商品页或官方活动页，让用户在来源页面查看最新图片、价格和库存。

## 内容版权规则

不得保存、缓存或展示外部网站的全文内容，包括：

- 活动说明全文；
- 商品说明全文；
- 新闻全文；
- 用户评论全文；
- 商品问答全文；
- 平台帮助页或规则页全文。

本站只保存必要事实信息，例如活动名称、活动时间、参加条件、返还上限、来源链接和更新时间，并使用原创中文说明帮助用户理解。

核心原则：只保存事实，不保存外部网站的原文表达。需要引用时只做短句必要引用，并优先改写为自己的说明。

## 数据来源优先级

按优先级从高到低使用：

1. 官方 API：例如楽天 API、Yahoo API。使用前必须阅读 API 条款、限额、署名和展示要求。
2. 官方活动页：例如平台活动页、店铺チラシ、官方キャンペーン页。
3. 官方商品页：仅少量人工核验，不批量抓取。
4. 官方 App / 店头海报 / 邮件通知：必须写明“需用户自行确认门店和结算页”。
5. 用户投稿或群友反馈：只能作为线索，不能直接写成已核验优惠。
6. 第三方网站：只作为参考，不作为最终优惠来源，除非能回到官方页面核验。

## 来源等级

`sourceLevel` 可选值：

- `official`：官方页面、官方活动页、官方商品页、官方店铺チラシ。
- `authorized`：官方 API、联盟计划、商家授权数据、明确授权素材。
- `community`：用户投稿、群友反馈、朋友提供的购物截图或经验。
- `signal`：社交平台、第三方网站、新闻、传闻线索、尚未回到官方来源核验的信息。
- `unknown`：来源不明或来源链条无法确认。

规则：

- 只有 `official` / `authorized` 可以标记为 `dataStatus: "verified"`。
- `community` / `signal` 只能作为线索，不能直接写成已核验优惠。
- `unknown` 不得写入已核验优惠，也不得作为页面推荐依据。
- 如果线索来自社区或第三方，必须回到官方页面、官方 API 或授权来源复核后，才可以升级为 `verified`。

## 已核验优惠写入规则

写入 `data/deals.json` 并标记为 `dataStatus: "verified"` 时，必须满足：

- 有 `url`，链接到官方活动页、官方商品页或可核验来源页。
- 有 `sourceName`，说明来源名称。
- 有 `sourceLevel`，且只能是 `official` 或 `authorized`。
- 有 `updatedAt`，使用实际核验日期。
- 有明确 `availabilityStatus`：`active`、`expired` 或 `unavailable`。
- 有 `participationSteps`，用简单中文说明怎么参加。
- 有 `savingsExample`，举例说明买多少大概能省多少。
- 有 `maxBenefitExample`，说明最大可省/可返场景和限制。
- 有 `checkBeforeBuying`，提醒用户下单前核价、看库存、看送料、看积分内訳。

如果缺少具体商品页、价格、活动条件或来源，只能：

- 保持 `dataStatus: "sample"`；
- 或记录到 `docs/latest-signals.md`；
- 或写入攻略/日历作为“购买时机提醒”，不能写成确定商品优惠。

## 日历活动写入规则

写入 `data/shopping-calendar.json` 时，必须尽量使用官方来源，并说明：

- 活动是什么；
- 什么时候参加；
- 怎么参加；
- 适合买什么；
- 下单前确认什么；
- 举例说明可能省多少钱；
- 说明最大返还或最大优惠的限制。

对抽选、地区限定、门店限定活动必须明确提示：

- 不是所有门店参加；
- 不是所有商品参加；
- 返还不一定必得；
- 最大返还不是稳定收益；
- 最终以官方页面、店头公告和结算页为准。

## 自动化边界

短期自动化只能做：

- 定时提醒人工查看官方活动；
- 读取本站已有 JSON/MDX 内容并做格式校验；
- 根据人工提供的信息更新内容；
- 使用官方 API 获取允许范围内的数据；
- 低频访问少量官方活动页用于核验，且不绕过限制。

不得让 Codex 或脚本生成以下行为：

```python
while True:
    requests.get("https://example-shopping-site/...")
```

也不得生成：

- 无限循环抓取；
- 多线程批量抓取；
- 模拟登录抓取；
- 自动破解验证码；
- 隐藏真实访问来源；
- 绕过 robots、访问限制或平台条款；
- 未经允许抓取并保存大量商品、评论、用户数据。

## 第二阶段 API 规则

如果后续使用楽天 API、Yahoo API 或其他官方 API：

- 先确认 API 条款允许当前用途。
- 遵守请求频率、缓存时间、展示要求和署名要求。
- 保存最少必要数据，不保存不需要的用户或平台数据。
- 对 API 返回的价格、库存、积分仍标注更新时间。
- API 失败时不自动编造数据。

## 联盟链接规则

如果页面包含联盟链接，必须明确标识。

规则：

- 不得因为佣金高而优先推荐商品。
- 不得隐藏联盟链接属性或误导用户以为链接来自本站自营。
- 不得夸大优惠、收益、返现或积分价值。
- 不得把积分返还、期间限定ポイント或抽选返还写成现金折扣。
- 推荐理由必须基于价格、适用场景、用户需求和来源核验，而不是佣金。
- 购买前仍必须提醒用户以官方页面、结算页面和门店公告为准。

Footer 或相关页面需要保留类似说明：

```txt
本站部分链接可能为联盟链接。购买前请以官方页面和结算页面为准。
```

## 第三阶段价格监控规则

只有在网站有真实用户、真实需求和清晰合规方案后，才考虑价格监控。

价格监控优先路径：

1. 官方 API。
2. 联盟计划或商家授权数据。
3. 用户手动记录常买价格。
4. 低频、少量、符合条款的公开页面核验。

不得以“技术能做到”为理由绕过平台限制。

## AI 生成内容规则

AI 可以协助整理、改写、排版、生成代码和做格式校验，但不得编造以下内容：

- 商品价格；
- 库存状态；
- 活动时间；
- 优惠条件；
- 积分倍率；
- 官方来源；
- 用户评价；
- 商品销量；
- 返还上限；
- 品牌授权关系。

如果来源不明确，只能写成“线索”或“待核验”，不能写成 `verified`。AI 生成的省钱示例必须基于已知规则做演算，并写明“约”“预计”“以结算页为准”等限制。

## 用户承诺与页面表达

页面文案必须保持谨慎：

- 用“预计”“约”“以结算页为准”“需自行确认”。
- 不写“必省”“稳赚”“全网最低”“一定返”。
- 对积分、期间限定ポイント、PayPayポイント等说明其限制。
- 对宝宝健康、安全、医疗相关内容只做信息整理，不替代专业建议。
- 页面 Footer 必须保留免责声明，说明本站只整理公开活动信息与购物攻略，价格、库存、积分规则可能变化，最终以官方页面、结算页面和门店公告为准。

## 每次工作前检查

涉及优惠、活动、价格、库存、自动化时，先检查：

1. 来源是否官方或可核验？
2. `sourceLevel` 是否足以支持 `verified`？
3. 是否涉及外部图片或外部全文内容？如果涉及，默认不保存。
4. 是否需要登录、验证码或绕过限制？如果需要，停止。
5. 是否在批量抓取？如果是，停止。
6. 是否可能误导用户以为一定能省钱？如果是，改成谨慎表达。
7. 是否写清参加方法、限制、更新时间和最终核价提醒？

如果不确定是否合规，默认不要自动化，先记录为线索或改成人工核验流程。
